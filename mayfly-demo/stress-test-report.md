## 三、监控与指标收集

### 3.1 监控架构

Mayfly 项目内置了完整的监控指标收集系统，基于 **Micrometer** 实现，支持多种监控后端（如 Prometheus、Grafana 等）。

#### 3.1.1 核心组件

| 组件 | 说明 | 实现类 |
|------|------|--------|
| **MetricsCollector 接口** | 监控指标收集器的标准接口 | `io.mayfly.monitor.MetricsCollector` |
| **MicrometerMetricsCollector** | 基于 Micrometer 的完整实现 | `io.mayfly.monitor.MicrometerMetricsCollector` |
| **NoOpMetricsCollector** | 空操作实现（监控未启用时使用） | `io.mayfly.monitor.NoOpMetricsCollector` |

#### 3.1.2 监控指标

**MicrometerMetricsCollector 收集的指标**：

| 指标名称 | 类型 | 说明 |
|---------|------|------|
| `mayfly.model.calls.total` | Counter | 模型调用总次数 |
| `mayfly.model.calls.success` | Counter | 成功调用次数 |
| `mayfly.model.calls.failure` | Counter | 失败调用次数 |
| `mayfly.model.calls.failover` | Counter | 故障转移次数 |
| `mayfly.model.latency.seconds` | Timer | 模型调用延迟 |
| `mayfly.model.tokens.input` | DistributionSummary | 输入 Token 数 |
| `mayfly.model.tokens.output` | DistributionSummary | 输出 Token 数 |

#### 3.1.3 自动配置

在 `MayflyAutoConfiguration` 中，监控收集器根据配置自动注入：

```java
@Bean
@ConditionalOnMissingBean
public MetricsCollector metricsCollector() {
    if (properties.getMonitor() != null && properties.getMonitor().isEnabled()) {
        // 当启用监控时，应返回 MicrometerMetricsCollector
        // 当前实现返回 NoOpMetricsCollector（待优化）
    }
    return new NoOpMetricsCollector();
}
```

**注意**：当前自动配置存在一个待优化问题 - 即使启用监控，也可能返回 `NoOpMetricsCollector`。在生产环境中，建议手动配置 `MicrometerMetricsCollector` Bean。

### 3.2 本次压测的监控方式

**本次压测使用了以下监控工具**：

| 工具 | 监控内容 | 说明 |
|------|---------|------|
| **VisualVM 2.2.1** | JVM 堆内存、活动线程数、GC 情况 | 实时图形化监控 |
| **Windows 任务管理器** | CPU 使用率、内存使用、线程数、句柄数 | 系统级监控 |
| **应用日志** | 错误日志、熔断器状态、故障转移记录 | 通过 `mayfly.log` 文件分析 |
| **JMeter 聚合报告** | 响应时间、吞吐量、错误率 | 压测工具内置报告 |

### 3.3 监控数据收集情况

**本次压测中，内置的 MetricsCollector 未启用**，原因如下：

1. **配置问题**：`application.yml` 中未显式启用监控模块
2. **自动配置限制**：当前 `MayflyAutoConfiguration` 的监控配置逻辑需要优化
3. **监控重点**：本次压测主要关注系统级指标（内存、线程、错误率），而非业务指标（Token 数、故障转移次数等）

**建议后续压测启用内置监控**：

```yaml
mayfly:
  monitor:
    enabled: true
```

并手动配置 `MicrometerMetricsCollector` Bean：

```java
@Bean
public MetricsCollector metricsCollector(MeterRegistry meterRegistry) {
    return new MicrometerMetricsCollector(meterRegistry);
}
```

## 七、重要说明：外部API限制

### 7.1 当前压测的限制因素

**本次压测结果受到外部AI模型API的严格限制**：

1. **API调用频率限制**：
   - 各模型厂商对API调用有每秒/每分钟调用次数限制
   - 当并发请求超过API限制时，会触发限流或拒绝服务
   - 这直接导致了高并发场景下的错误率上升

2. **API响应时间**：
   - AI模型API的响应时间通常在5-20秒之间
   - 大量并发请求会导致API响应队列堆积
   - 这限制了系统的实际吞吐能力

3. **API可用性**：
   - 智谱AI模型在测试中频繁触发熔断器
   - 说明该模型的API稳定性或配额可能不足
   - 这影响了整体成功率

### 7.2 如果外部API限制解除

**如果大模型厂商的每秒调用次数上调，或者接入更多API密钥，系统的并发能力暂时无法预估**：

1. **理论并发能力**：
   - 当前测试中，300并发和500并发的活动线程数几乎相同（222 vs 221）
   - 这说明**本地系统的线程处理能力远未达到极限**
   - 如果外部API不再成为瓶颈，系统可能支持更高的并发数

2. **潜在瓶颈转移**：
   - 如果API限制解除，瓶颈可能转移到：
     - 本地CPU处理能力
     - 本地内存容量
     - 网络带宽
     - JVM线程池配置

3. **建议的后续测试**：
   - 接入更多API密钥（轮询使用）
   - 使用Mock服务模拟API响应（消除外部依赖）
   - 进行纯本地性能测试（不依赖外部API）
   - 逐步增加并发数，找到本地系统的真实极限

### 7.3 生产环境建议

1. **API密钥池化**：
   - 为每个模型配置多个API密钥
   - 实现密钥轮询机制，分散调用压力
   - 避免单个密钥触发限流

2. **API限流适配**：
   - 根据各厂商的限流规则，配置合适的并发数
   - 实现动态限流调整机制
   - 监控API调用频率，避免触发限流

3. **多厂商冗余**：
   - 接入更多AI模型厂商作为备用
   - 当某厂商API不可用时，自动切换到其他厂商
   - 提高系统整体可用性

## 八、优化建议

### 8.1 短期优化（立即实施）

1. **调整模型权重**：
   - 当前配置：DeepSeek 60%、通义千问 25%、DeepSeek Backup 10%、智谱AI 5%
   - 建议：根据实际API稳定性调整权重
   - 智谱AI错误率较高，可进一步降低权重或暂时禁用

2. **优化Ramp-Up时间**：
   - 场景3（200并发，Ramp-Up 30秒）表现最优
   - 建议生产环境使用平缓的Ramp-Up策略
   - 避免瞬间高并发冲击

3. **监控告警配置**：
   - 配置错误率告警（阈值：5%）
   - 配置内存使用告警（阈值：80%）
   - 配置线程数告警（阈值：200）

### 8.2 中期优化（1-2周）

1. **API密钥池化**：
   - 为每个模型配置3-5个API密钥
   - 实现密钥轮询机制
   - 降低单个密钥的调用频率

2. **熔断器参数优化**：
   - 根据实际API响应时间调整熔断器阈值
   - 优化熔断器恢复策略
   - 减少不必要的熔断触发

3. **连接池优化**：
   - 配置HTTP连接池参数
   - 优化连接复用策略
   - 减少连接建立开销

### 8.3 长期优化（1-2月）

1. **异步处理机制**：
   - 引入消息队列（如RabbitMQ、Kafka）
   - 实现请求异步处理
   - 提高系统吞吐量

2. **缓存机制**：
   - 对相似请求实现缓存
   - 减少重复的API调用
   - 降低外部API依赖

3. **分布式部署**：
   - 实现多实例部署
   - 配置负载均衡
   - 提高系统整体并发能力
